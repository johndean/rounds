## 1. Why v2 supersedes v1

v1 was scoped from the user's verbal sketch. /autoplan review against the actual schema surfaced 5 critical schema-reality mismatches + 4 high-severity issues + 4 medium + 4 low. The fixes are mechanical but pervasive — every SQL fragment in v1 was wrong. v2 rebuilds against verified migration state (mig 001, 020, 022, 025, 034, 036) and incorporates the user's 3 gate decisions (combined PR, structural inverse undo, keep /corrections shape).

**Three user decisions that shape v2:**
1. **Combined deliverable.** Phase 3.5 + Phase 4 ship in one PR behind `SPLIT_MERGE_ENABLED=false`. No dead-code window.
2. **Structural inverse undo.** `split` correction stores invert metadata; `merge` stores re-split metadata. Real round-trip undo/redo.
3. **Keep `/v1/corrections {correction_type: split|merge}` endpoint shape.** Parent plan honored.

---

## 2. Problem statement

Today `app/api/corrections.py:46-50` whitelists `'split'` and `'merge'` as legal `correction_type` values, and the route at `POST /v1/sessions/{sid}/corrections` 200s on either type — but **no executor handles them**. The correction lands in `correction_ledger`, the pointer advances, and `segments` is untouched.

The split UI work in the parent plan (`SegmentText.vue` right-click split-here + merge-up keystroke) is currently a silent no-op against this backend. v2 ships the backend executor AND the UI wiring as one atomic change.

## 3. Goals + non-goals

**Goals.**
1. Make `POST /v1/corrections {correction_type: 'split', after_word_index: N}` actually split a segment into two adjacent rows.
2. Make `POST /v1/corrections {correction_type: 'merge', expected_right_segment_id: <UUID>}` actually merge two adjacent same-speaker segments into one.
3. Preserve `word_alignment` integrity with `gemini_idx` resequenced 0..N-1 in the new segment.
4. Define the `key_points_annotations` clone-to-both-halves policy.
5. Inherit `slide_id`, `flags JSONB`, `metadata JSONB` correctly on split.
6. Reject `is_anchor=TRUE` segments (poll/chat anchors are atomic UI blocks).
7. Serialize concurrent split/merge ops on the same session via Postgres advisory lock from Phase 1.5.
8. Resolve the autosave-vs-split race by re-reading `segments.text` AFTER `SELECT FOR UPDATE` inside the lock.
9. **Structural inverse undo** — split correction stores enough metadata to be undone by merging the two halves; merge stores enough to re-split. Undo/redo round-trip leaves `segments` byte-identical.
10. Ship Phase 4's split UI (`SegmentText.vue` cursor → split here; merge-up keystroke) wired to this backend.
11. Gate the combined deliverable behind `SPLIT_MERGE_ENABLED` env flag, default `false`.

**Non-goals.**
- No migration (mig 020's `(session_id, content_hash)` UNIQUE + mig 022's seq index changes already in place).
- No changes to locked processing weights (`FUSION_*`, `ALIGN_*`, `IIL_*`, `CELERY_*`).
- No changes to existing 11 `correction_type` handlers — dispatcher only adds a pre-branch.
- No re-running of `lcs_discrepancies_task` from inside the executor. After-split alignments/words/discrepancies/normalization_results may be stale — recovery is a manual `/v1/diag/realign/<sid>` curl (documented in the verification checklist).
- No find-replace cross-segment splits (out of scope; existing find-replace is segment-local).
- No mobile UI — Phase 4 desktop only.

### 4.2 Dispatcher in `corrections.py`

Pre-branch in `apply_correction` BEFORE the existing ledger-append path. The 11 other correction types are untouched.

```python
# After validation, before the existing _is_noop_correction / _truncate_redo_tail path:
if body.correction_type in {"split", "merge"}:
    if not settings.SPLIT_MERGE_ENABLED:
        raise HTTPException(503, detail={"code": "SPLIT_MERGE_DISABLED"})
    async with try_advisory_lock_async(db, session_id=sid, stage="split_merge") as acquired:
        if not acquired:
            raise HTTPException(409, detail={"code": "SPLIT_MERGE_BUSY"})
        # action_id-based dedup inside the lock
        if await _has_existing_action(db, sid, body.action_id):
            return await _replay_existing_response(db, sid, body.action_id)
        if body.correction_type == "split":
            result = await segment_split.execute_split(db, sid, body, user)
        else:
            result = await segment_merge.execute_merge(db, sid, body, user)
        # Append ledger row with the invert metadata embedded in new_text (JSON)
        # then commit and emit WS event.
        ledger_row = await _append_split_merge_ledger(db, sid, body, result, user)
        await db.commit()
        await _emit_ws(sid, {
            "type": "correction_applied",
            "action_id": str(ledger_row["action_id"]),
            "segment_ids": result["affected_segment_ids"],
            "correction_type": body.correction_type,
        })
        return _serialize_response(ledger_row, result)
```

The advisory lock is per-`(session_id, "split_merge")`, separate from any per-segment autosave lock Phase 2 may use, so an autosave on segment X does not deadlock with a split on segment Y.

**Autosave-race fix.** Inside the executor, after `SELECT … FOR UPDATE` on the target segment row, **re-read** `segments.text` from that locked row before computing the split. This catches any autosave UPDATE that landed between the API request and the lock acquisition. The Phase 2 autosave's bare `UPDATE segments SET text = :t` at `corrections.py:317-325` cannot race past `FOR UPDATE` because Postgres serializes through the row lock.

### 4.3 Split executor (`app/services/segment_split.py`)

**Input** (`SplitBody`, extends `CorrectionRequest`):
```python
class SplitBody(BaseModel):
    correction_type: Literal["split"]
    segment_id: UUID
    after_word_index: int = Field(..., ge=0)   # split AFTER this word index (0-based)
    action_id: Optional[UUID] = None
```

**Why word-index, not char-offset.** `word_alignment` keys on `(segment_id, gemini_idx)` where `gemini_idx` is the 0-based position in `seg.text.split()`. The product invariant is "split between two words at a known timing boundary." char_offset would require word-boundary reasoning every call; `after_word_index` is the natural primitive. The frontend converts cursor position → nearest word span → `after_word_index` once at submit time.

**Algorithm.**

1. **Lock + re-read.**
   ```sql
   SELECT id, session_id, slide_id, speaker_id, seq, start_ms, end_ms,
          text, flags, is_anchor, anchor_kind, metadata, content_hash
     FROM segments
    WHERE id = :sid AND session_id = :session_id
    FOR UPDATE
   ```
   Reject `SPLIT_SEGMENT_NOT_FOUND` if no row.

2. **Reject anchors.** If `is_anchor = TRUE`, raise `SPLIT_ANCHOR_SEGMENT` — poll/chat anchor segments are atomic UI blocks.

3. **Resolve word boundary.** Load `word_alignment` rows for this segment ordered by `gemini_idx`:
   ```sql
   SELECT gemini_idx, stt_start_ms, stt_end_ms, stt_word_id, match_kind
     FROM word_alignment
    WHERE segment_id = :sid
    ORDER BY gemini_idx
   ```
   Reject `SPLIT_INVALID_WORD_INDEX` if `after_word_index < 0` or `after_word_index >= n_words - 1` (must leave ≥1 word in each half).

4. **Compute split point.**
   - `words_a = words[0 .. after_word_index]` (inclusive), `words_b = words[after_word_index+1 ..]`.
   - `split_ms = words_b[0].stt_start_ms if not NULL else words_a[-1].stt_end_ms + 1`.
   - If both are NULL (`match_kind = 'unmatched'` on both sides), fall back to proportional: `split_ms = start_ms + (end_ms - start_ms) * (after_word_index + 1) / n_words`.
   - `text_a = " ".join(words tokens[:after_word_index+1])`, `text_b = " ".join(tokens[after_word_index+1:])`.
   - Use the same word tokenization as `lcs_discrepancies_task` to ensure `gemini_idx` alignment stays consistent.

5. **Compute new content_hash for the new half.** `new_hash = encode(sha256((session_id || split_ms)::bytea), 'hex')`. Original half's hash is NOT recomputed — `start_ms` unchanged, hash unchanged.

6. **UPDATE original segment.**
   ```sql
   UPDATE segments
      SET text = :text_a, end_ms = :split_ms, updated_at = now()
    WHERE id = :sid
   ```
   (`content_hash`, `seq`, `slide_id`, `speaker_id`, `flags`, `metadata` all unchanged.)

7. **INSERT new segment.**
   ```sql
   INSERT INTO segments (
       id, session_id, slide_id, speaker_id, seq,
       start_ms, end_ms, text, confidence,
       flags, is_anchor, anchor_kind, metadata, content_hash
   ) VALUES (
       gen_random_uuid(), :session_id, :slide_id, :speaker_id, :seq_b,
       :split_ms, :end_ms, :text_b, :confidence,
       :flags, FALSE, NULL, :metadata, :new_hash
   ) RETURNING id
   ```
   Where:
   - `seq_b = original.seq + 0.5` rounded to next free seq (Postgres `seq` is integer; we use `original.seq + 1` and renumber later segments only if a collision exists). **Note:** mig 022 dropped `UNIQUE (session_id, seq)`, so we don't strictly need uniqueness — but ordering by `seq` then `start_ms` keeps the editor's segment order stable.
   - Practical implementation: `seq_b = (SELECT COALESCE(MIN(seq), original.seq + 2) FROM segments WHERE session_id = :sid AND seq > original.seq) - 1`. If that produces a value `<= original.seq`, fall back to `original.seq + 1` and shift later rows by 1.
   - `slide_id`, `speaker_id`, `flags` inherited verbatim.
   - `metadata = original.metadata || jsonb_build_object('split_from', :original_id, 'split_at_ms', :split_ms)` — append a breadcrumb.
   - `is_anchor = FALSE`, `anchor_kind = NULL` — a new segment created by split is never an anchor.

8. **Reparent `word_alignment`.**
   ```sql
   UPDATE word_alignment
      SET segment_id = :new_id,
          gemini_idx = gemini_idx - :first_moved_idx
    WHERE segment_id = :original_id
      AND gemini_idx > :after_word_index
   ```
   where `:first_moved_idx = after_word_index + 1`. This resequences gemini_idx 0..N-1 for the new segment.

9. **`key_points_annotations` clone policy.**
   ```sql
   INSERT INTO key_points_annotations (
       id, session_id, segment_id, key_points, explanation,
       available, extraction_confidence, label, score, created_at
   )
   SELECT gen_random_uuid(), session_id, :new_id, key_points, explanation,
          available, extraction_confidence, label, score, now()
     FROM key_points_annotations
    WHERE segment_id = :original_id
   ```
   Both halves end up with the same `key_points` JSONB array. A subsequent `kp_task` re-run will refine. No data loss; UNIQUE (session_id, segment_id) holds because new_id is new.

10. **Compute structural-inverse metadata for the ledger.** Store as JSON in `correction_ledger.new_text` (repurposed for split/merge to carry invert payload):
    ```json
    {
      "kind": "split_invert",
      "original_segment_id": "<uuid>",
      "new_segment_id": "<uuid>",
      "split_at_word_index": <int>,
      "split_at_ms": <int>,
      "original_text_before": "<full original text>",
      "original_end_ms_before": <int>,
      "original_content_hash_before": "<hex>",
      "moved_word_alignment_count": <int>,
      "kp_cloned": true
    }
    ```

11. **Return** to dispatcher:
    ```python
    return {
        "affected_segment_ids": [str(original_id), str(new_id)],
        "invert_payload": <JSON above>,
    }
    ```

### 4.4 Merge executor (`app/services/segment_merge.py`)

**Input** (`MergeBody`):
```python
class MergeBody(BaseModel):
    correction_type: Literal["merge"]
    segment_id: UUID                                # the LEFT segment, kept after merge
    expected_right_segment_id: UUID                 # frontend echoes the right neighbor
    action_id: Optional[UUID] = None
```

**Algorithm.**

1. **Lock LEFT.**
   ```sql
   SELECT * FROM segments WHERE id = :left_id AND session_id = :sid FOR UPDATE
   ```
   Reject `MERGE_LEFT_NOT_FOUND`. Reject `MERGE_ANCHOR_SEGMENT` if `is_anchor`.

2. **Find right neighbor.**
   ```sql
   SELECT * FROM segments
    WHERE session_id = :sid AND seq > :left_seq
    ORDER BY seq, start_ms
    LIMIT 1
    FOR UPDATE
   ```
   Reject `MERGE_NO_NEIGHBOR` if no row. Reject `MERGE_NEIGHBOR_CHANGED` if `right.id != expected_right_segment_id`.

3. **Speaker check.** Reject `MERGE_SPEAKER_MISMATCH` if `left.speaker_id != right.speaker_id`. (No implicit cross-speaker merge.)

4. **Slide check.** Soft warn (not reject) if `left.slide_id != right.slide_id` — the merged segment keeps `left.slide_id`. Log a row to `audit_events` `kind='merge.slide_mismatch'` with both ids.

5. **Anchor check.** Reject `MERGE_ANCHOR_NEIGHBOR` if `right.is_anchor`.

6. **Build merged state.**
   - `merged_text = left.text.rstrip() + " " + right.text.lstrip()`.
   - `merged_end_ms = right.end_ms` (left's `start_ms` and `content_hash` unchanged).
   - `merged_flags = left.flags + (right.flags - left.flags deduped)` — union of JSONB arrays.
   - `merged_metadata = left.metadata || jsonb_build_object('merged_from', right.id, 'merged_at_ms', right.start_ms)`.

7. **UPDATE left.**
   ```sql
   UPDATE segments
      SET text = :merged_text, end_ms = :merged_end_ms,
          flags = :merged_flags, metadata = :merged_metadata, updated_at = now()
    WHERE id = :left_id
   ```

8. **Reparent word_alignment.** Right's rows move to left, with `gemini_idx` shifted by left's word count.
   ```sql
   WITH left_count AS (
       SELECT COALESCE(MAX(gemini_idx), -1) + 1 AS n FROM word_alignment WHERE segment_id = :left_id
   )
   UPDATE word_alignment
      SET segment_id = :left_id,
          gemini_idx = gemini_idx + (SELECT n FROM left_count)
    WHERE segment_id = :right_id
   ```

9. **`key_points_annotations` merge policy.** If both segments have rows: concatenate `key_points` arrays (max 5 — drop overflow), keep `left.explanation` (right's discarded), `available = TRUE if either`, `extraction_confidence = max`. If only one has a row: leave the existing row on left, delete right's. Single UPSERT via `ON CONFLICT (session_id, segment_id) DO UPDATE`.

10. **DELETE right.**
    ```sql
    DELETE FROM segments WHERE id = :right_id
    ```
    Cascade handles `alignments`, `transcription_discrepancies`, `normalization_results`, `words` rows FK-pointed at right. **Operator note:** post-merge alignments referencing the deleted right segment are gone; re-run `/v1/diag/realign/<sid>` to repopulate.

11. **Compute invert metadata.**
    ```json
    {
      "kind": "merge_invert",
      "kept_segment_id": "<left_id>",
      "deleted_segment_id": "<right_id>",
      "left_text_before": "<left.text pre-merge>",
      "left_end_ms_before": <int>,
      "left_flags_before": <jsonb>,
      "left_metadata_before": <jsonb>,
      "right_full_row": {
         "id": "<right_id>", "session_id": "<sid>", "slide_id": "<right.slide_id>",
         "speaker_id": "<right.speaker_id>", "seq": <int>, "start_ms": <int>, "end_ms": <int>,
         "text": "<right.text>", "confidence": <float>, "flags": <jsonb>,
         "is_anchor": false, "anchor_kind": null, "metadata": <jsonb>,
         "content_hash": "<hex>"
      },
      "right_word_alignment": [<full alignment rows pre-merge>],
      "right_kp_row": <full kp row or null>
    }
    ```

12. **Return.**
    ```python
    return {
        "affected_segment_ids": [str(left_id)],
        "deleted_segment_id": str(right_id),
        "invert_payload": <JSON above>,
    }
    ```

### 4.5 Structural inverse (undo/redo)

The existing `_materialize_segments_for_session(db, sid, *, up_to_pointer)` in `corrections.py:173` ONLY replays `text_edit` corrections. v2 extends it (or adds a sibling) to handle split/merge inverts.

**On `undo_correction` and `redo_correction`,** after the pointer moves to `new_ptr`, walk corrections from sequence `min(old_ptr, new_ptr) + 1` to `max(old_ptr, new_ptr)`:
- For each row, if `correction_type in ('split', 'merge')`, parse the `invert_payload` from `new_text`.
- If undoing (new_ptr < old_ptr), apply the INVERSE operation (split → merge halves; merge → re-split right back).
- If redoing (new_ptr > old_ptr), apply the FORWARD operation again (re-execute the split or merge from the stored payload).

**Inverse algorithms.**

`invert_split(db, payload)`:
1. UPDATE segments SET text = original_text_before, end_ms = original_end_ms_before WHERE id = original_segment_id.
2. UPDATE word_alignment SET segment_id = original_segment_id, gemini_idx = gemini_idx + (original word count) WHERE segment_id = new_segment_id.
3. DELETE FROM key_points_annotations WHERE segment_id = new_segment_id (cloned row).
4. DELETE FROM segments WHERE id = new_segment_id.

`invert_merge(db, payload)`:
1. INSERT INTO segments using right_full_row.
2. UPDATE segments SET text = left_text_before, end_ms = left_end_ms_before, flags = left_flags_before, metadata = left_metadata_before WHERE id = kept_segment_id.
3. Move right's word_alignment rows back: UPDATE word_alignment SET segment_id = deleted_segment_id, gemini_idx = gemini_idx - (left word count) WHERE segment_id = kept_segment_id AND gemini_idx >= (left word count).
4. INSERT right_kp_row (if non-null) into key_points_annotations.

Both inverse ops use the same `try_advisory_lock_async(stage="split_merge")` to serialize against forward ops.

## 5. Test plan (30 tests)

### 5.1 `tests/test_segment_split.py` (14 tests)

1. Happy path: split a 4-word segment after word 1 → 2 segments, words correctly resequenced (gemini_idx 0..1 on left, 0..1 on right).
2. `after_word_index = 0` → 2 segments, left has 1 word, right has 3.
3. `after_word_index = n-1` → 400 `SPLIT_INVALID_WORD_INDEX` (right would have 0 words).
4. `after_word_index < 0` → 400.
5. `after_word_index >= n_words - 1` → 400.
6. Anchor segment (`is_anchor=TRUE`) → 400 `SPLIT_ANCHOR_SEGMENT`.
7. Segment with no `word_alignment` rows → 422 `SPLIT_NO_WORD_ALIGNMENT`.
8. `key_points_annotations` row present → cloned to new segment (both rows have same `key_points` array).
9. `flags` JSONB inherited by new segment.
10. `metadata` on new segment has `split_from` + `split_at_ms` breadcrumb.
11. `slide_id` + `speaker_id` inherited by new segment.
12. `content_hash` recomputed for new segment (`sha256(session_id||split_ms)`); original's hash unchanged.
13. Duplicate `action_id` → idempotent (only one new segment created across two POSTs, cached response returned on the second).
14. **Concurrent splits** on same session: second request returns 409 `SPLIT_MERGE_BUSY` while first holds lock.

### 5.2 `tests/test_segment_merge.py` (10 tests)

1. Happy path: merge two adjacent same-speaker segments → 1 row, text concat with single space, word_alignment shifted.
2. Merge last segment (no right neighbor) → 400 `MERGE_NO_NEIGHBOR`.
3. Cross-speaker → 400 `MERGE_SPEAKER_MISMATCH`.
4. `expected_right_segment_id` mismatch → 409 `MERGE_NEIGHBOR_CHANGED`.
5. Left is anchor → 400 `MERGE_ANCHOR_SEGMENT`.
6. Right is anchor → 400 `MERGE_ANCHOR_NEIGHBOR`.
7. Both sides have `key_points_annotations` → concat capped at 5 entries, max confidence preserved.
8. Slide mismatch → succeeds, `audit_events` row written.
9. `flags` JSONB union (no duplicates).
10. `metadata` has `merged_from` + `merged_at_ms` breadcrumb.

### 5.3 `tests/test_segment_inverse.py` (6 round-trip tests)

1. Split → undo → segments byte-identical to pre-split state (text, end_ms, content_hash, word_alignment, kp_annotations).
2. Split → undo → redo → state matches first split result.
3. Merge → undo → segments byte-identical to pre-merge state.
4. Merge → undo → redo → state matches first merge result.
5. Split → text_edit on left half → undo (the text_edit) → undo (the split) → original state recovered.
6. Multiple splits → multiple undos in reverse order → state recovered at each step.

## 7. Rollback

1. **Flag flip.** Set `SPLIT_MERGE_ENABLED=false` on Railway. Backend returns 503 immediately; frontend hides menu items on next page load.
2. **Code revert.** `git revert <merge-sha>`. The two service files are new, dispatcher pre-branch is a single block, no migration to back out, no data to clean up (no users have hit the executor with the flag off).
3. **Data recovery if undo fails post-rollback.** If a split or merge landed with the flag on and rollback wipes the inverse logic, the data is still queryable — the ledger row carries the invert payload in `new_text`. A one-off script could replay inverses if needed. **Not anticipated; documented for completeness.**
