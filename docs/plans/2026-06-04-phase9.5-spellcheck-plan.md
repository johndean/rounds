# Phase 9.5 — Spellcheck Implementation Plan

**Generated**: 2026-06-04
**Predicate**: Phase 9 research (`docs/research/spellcheck-grammarly-2026-06-04.md`) found Grammarly Embedded SDK shut down 2024-01-10. Self-hosted LanguageTool + Hunspell medical dictionary is the recommended replacement.
**Authorization**: User accepted recommended stack defaults during Phase 1+8+9 close-out (2026-06-04). Plan is now a working doc, not a research deliverable.

## Stakeholder defaults (accepted as planning premise)

| Decision | Accepted default |
|---|---|
| BAA with Anthropic/Google | Not needed — PHI never leaves rounds infra in the implementation below. Only the optional LLM polish layer would touch external; that layer is OPT-IN per user. |
| GPLv3 medical dictionary | Used as data, not code. License-safe; ship as a bundled asset. |
| UX target | Inline red squiggles (native `spellcheck="true"` style) + a sidebar "issues" list for grammar/style flags from LanguageTool. |
| Latency budget | 500-800ms acceptable (server roundtrip on debounce). Client-side spellcheck still fires immediately. |
| Scale assumption | ~10 editors, ~50M chars/mo. Confirmed during decision capture. |

## Stack

Four layers, additive. Each layer ships as its own commit and can be feature-flagged off independently.

```
Layer 1  ─────  Browser-native spellcheck (red squiggle)
Layer 2  ─────  + nspell + Hunspell medical dictionary client-side
Layer 3  ─────  + LanguageTool Docker service (server-side grammar/style)
Layer 4  ─────  + optional Claude Haiku "Polish segment" button (opt-in)
```

## Layer 1 — Browser-native spellcheck

Smallest possible win — make sure every `contenteditable` in TranscriptPane / chat / poll edit / etc. has `spellcheck="true"` set. Browser does its own red-squiggle pass; zero infra, zero cost, zero PHI risk.

**Scope**:
- Audit `frontend/src/components/editor/` for any `contenteditable` element with `spellcheck="false"` or unset (default depends on browser).
- Set `spellcheck="true"` explicitly on every text-editing surface.

**Files** (estimated):
- `frontend/src/components/editor/TranscriptPane.vue`
- `frontend/src/components/editor/PollsTab.vue`
- `frontend/src/components/editor/ChatTab.vue`
- `frontend/src/components/editor/SpeakerEditPanel.vue`
- any other contenteditable surface uncovered during audit

**Dependencies**: zero new.
**Tests**: Playwright check — type a known misspelling, expect `:misspelled` DOM state or a screenshot check.
**Rollback**: revert the commit. Trivial.
**Cost**: $0.
**Latency**: instant (in-process browser pass).
**Risk**: LOW. Cosmetic-level change.

## Layer 2 — Client-side Hunspell medical dictionary

Browser-native spellcheck uses the OS dictionary, which doesn't know veterinary / medical terminology. Add `nspell` + a Hunspell `.aff`/`.dic` pair for the medical domain so terms like "intracytoplasmic" or species names stop showing as misspelled.

**Scope**:
- Add `nspell` (~50KB) + Hunspell medical dictionary asset (~150KB unminified) to `frontend/`.
- New composable `frontend/src/composables/useMedicalSpellcheck.ts` that wraps nspell with a debounced API.
- Wire into TranscriptPane: paint custom underlines for words flagged by nspell but NOT in the medical dict.
- Mute browser-native spellcheck for those specific spans to avoid double-marking.

**Files**:
- `frontend/package.json` — add `nspell` dep
- `frontend/public/dict/en-med.aff` + `en-med.dic` — bundled assets (GPLv3-licensed data, shipped verbatim)
- `frontend/src/composables/useMedicalSpellcheck.ts`
- `frontend/src/components/editor/TranscriptPane.vue` — wire composable

**Dependencies**: 1 new (`nspell`).
**Tests**: unit test the composable with known-good + known-bad words (e.g. "histopathology" ✓, "histopathologie" ✗).
**Rollback**: feature flag `VITE_SPELLCHECK_MEDICAL_ENABLED=false` + revert.
**Cost**: $0 (client-side).
**Latency**: <50ms per word check.
**Risk**: LOW-MEDIUM. License: GPLv3 dictionary used as DATA (not linked into code) — legal-safe per common interpretation. Confirm with legal before shipping to production.

## Layer 3 — LanguageTool Docker service (server-side)

LanguageTool catches grammar and style issues nspell can't (subject-verb agreement, passive voice, "their/there/they're"). Self-hosted = PHI stays on rounds infra.

**Scope**:
- New Railway service: `languagetool` running official LanguageTool Docker image (`erikvl87/languagetool`).
- New backend route `POST /v1/spellcheck/check` that proxies to the LanguageTool container, returns structured issues.
- New composable `frontend/src/composables/useLanguageToolCheck.ts` — debounced 600ms, fires on segment blur or pause-in-typing.
- TranscriptPane sidebar: list of grammar/style flags with "ignore" / "accept" actions.

**Files**:
- `infra/languagetool/Dockerfile` — pinned LanguageTool version
- `infra/railway.json` — add service definition + memory/CPU caps
- `app/api/spellcheck.py` — new module with `/check` route
- `app/main.py` — register router
- `frontend/src/composables/useLanguageToolCheck.ts`
- `frontend/src/components/editor/TranscriptPane.vue` — wire sidebar
- `frontend/src/components/editor/SpellcheckSidebar.vue` — new component

**Dependencies**: 1 new Railway service. No new Python deps (just HTTP call via `httpx` which is already installed).
**Tests**:
- Backend: `tests/test_spellcheck_route.py` — happy path + LanguageTool-down fallback.
- E2E: Playwright — type known-bad sentence, expect grammar flag.
**Rollback**: env flag `SPELLCHECK_LT_ENABLED=false` on backend. Frontend gracefully shows "grammar checks unavailable" + degrades to Layers 1+2.
**Cost**: $5-15/mo (Railway micro instance for LT container).
**Latency**: 500-800ms per check (matches accepted default).
**Risk**: MEDIUM. New prod service = new failure mode. Mitigations:
  - Hard timeout 2s on `/v1/spellcheck/check`
  - Frontend falls back to "checks paused" message on timeout (no editor hang)
  - Health endpoint `/v1/spellcheck/health` polled every 60s; broken service goes quiet, doesn't block editing.

## Layer 4 — Optional Claude Haiku "Polish segment" button

OPT-IN. Editor selects a segment, clicks "Polish", Claude rewrites for clarity while preserving meaning. NOT triggered automatically. NOT applied to every segment.

**Scope**:
- New backend route `POST /v1/segments/{id}/polish` — calls Claude API with the segment text + a strict system prompt ("rewrite for clarity; do not change meaning; return ONLY the rewritten text").
- New button in TranscriptPane segment header.
- Preview mode — show diff, accept/reject.

**Files**:
- `app/api/polish.py` — new module
- `app/services/claude_client.py` — new helper (assumes `ANTHROPIC_API_KEY` env var)
- `frontend/src/composables/usePolish.ts`
- `frontend/src/components/editor/PolishDiff.vue` — diff preview

**Dependencies**: 1 new Python dep (`anthropic` SDK) OR direct httpx call.
**Tests**:
- Backend: `tests/test_polish_route.py` — mock Anthropic response.
- Manual smoke: polish a known-bad segment, verify diff renders.
**Rollback**: env flag `POLISH_ENABLED=false` (default OFF). Button hides; route returns 404.
**Cost**: ~$10-25/mo at 50M chars/mo scale (Claude Haiku at $0.25/M input + $1.25/M output, assuming ~5k polish calls/mo at ~500 char segments).
**Latency**: 1-3s per polish call. Async — UI shows spinner; doesn't block editing.
**Risk**: MEDIUM. The PHI consideration: this is the ONLY layer where session content leaves rounds infra. Mitigations:
  - Button is OPT-IN per segment, not bulk-applied.
  - Confirm with stakeholder BEFORE deploying Layer 4 if any session content is PHI-classified.
  - Alternative: route through a self-hosted local model (out of scope for this plan; defer to Phase 9.6 if Layer 4 is rejected).

## Sequencing recommendation

Ship the layers in order. Each layer's value is independent — even if Layer 4 never ships, Layers 1-3 deliver real editor-productivity wins.

**MVP for stakeholder demo** = Layer 1 + Layer 2. Zero infra cost, zero PHI risk, ~1-day implementation. Ship first.

**Production-grade** = Layer 1 + 2 + 3. ~3-5 days. Adds grammar/style coverage.

**Optional polish** = Layer 4. ~2 days. Requires PHI sign-off.

## Open risks

1. **GPLv3 dictionary license** — confirm with legal that bundling a GPL-licensed dictionary file (data, not code) does NOT virally license the rest of the rounds codebase. Standard interpretation: data files distributed alongside a non-GPL application do not contaminate the application. But legal sign-off should be on record.
2. **LanguageTool Docker resource use** — LT JVM is memory-hungry (~500MB minimum). Railway micro instance may not be enough; may need to scale to small instance ($10-15/mo). Validate with load test before committing.
3. **Polish layer (Layer 4) PHI** — confirm whether any veterinary session content crosses into PHI territory. If yes, defer Layer 4 until BAA or self-hosted LLM available.
4. **Browser fragmentation** — Safari's spellcheck behavior differs slightly from Chrome/Firefox. Test on Safari before claiming Layer 1 done.
5. **Performance at scale** — 50M chars/mo = ~1.7M chars/day. LanguageTool can comfortably handle this on a single micro instance, but spikes during peak edit hours (per-second concurrent users) should be monitored.

## Out of scope for Phase 9.5

- Speech-to-text correction (already handled upstream by transcription pipeline)
- Multi-language spellcheck (en-US only for now)
- Custom user dictionaries per editor (defer to Phase 9.6 if requested)
- Suggestions for medical terminology (Hunspell dict provides spell-correctness; no auto-complete)
