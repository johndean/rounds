## 1. Problem statement

`app/api/corrections.py:47` whitelists `'split'` and `'merge'` as legal `correction_type` values, so the API will 200 OK on payloads of those types — but **no executor handles them**. The correction is appended to the `corrections` ledger and that's it; `segments` is never mutated.

Today this is invisible because there is no caller — no frontend button, no migration, nothing that emits `correction_type='split'` or `'merge'`. **Phase 4** is about to ship the split UI on `SegmentText.vue`. Without 3.5, that UI is a silent no-op: the user clicks "Split here," sees a toast, and reloads to find the segment unchanged.

Phase 3.5 builds the real executors so Phase 4 has a backend to call.

## 2. Goals + non-goals

**Goals.**
1. Make `POST /v1/corrections` with `correction_type='split'` actually create a second `segments` row.
2. Make `POST /v1/corrections` with `correction_type='merge'` actually delete the second row and concat into the first.
3. Preserve `word_alignment` integrity across both ops.
4. Define and implement the `kp_annotations` (key-point anchors) policy on split.
5. Stay idempotent under retry/replay using existing `pg_try_advisory_lock` infra (Phase 1.5).
6. Cover every branch with tests — no production caller exists yet, so tests are the only check before Phase 4 lands.

**Non-goals.**
- No frontend changes (that's Phase 4).
- No migration (rely on existing `(session_id, content_hash)` unique index from migration 020).
- No undo/redo wiring beyond what materialization already does (Phase 2a).
- No changes to locked processing weights (FUSION_*, ALIGN_*, IIL_*, CELERY_*).
- No new error envelope codes; reuse the existing `CORRECTION_*` family.

### 3.2 Dispatcher (`app/api/corrections.py`)

Existing flow:
```python
if correction_type not in {'replace', 'insert', 'delete', 'split', 'merge'}:
    raise HTTPException(400, "...")
# append to corrections ledger
# return ledger row
```

New flow:
```python
async with try_advisory_lock_async(db, session_id, stage="split_merge"):
    if correction_type == 'split':
        result = await segment_split.execute(db, session_id, body, user_email)
    elif correction_type == 'merge':
        result = await segment_merge.execute(db, session_id, body, user_email)
    else:
        result = await _existing_replace_insert_delete_path(...)
# append to corrections ledger (after success)
# return ledger row enriched with result.new_segment_ids
```

The advisory lock is per-`(session_id, "split_merge")` — independent of the per-segment locks autosave uses — so a writer mid-autosave doesn't deadlock with a writer mid-split.

### 3.3 Split executor (`app/services/segment_split.py`)

**Input** (`SplitBody`):
```python
{
    "correction_type": "split",
    "segment_id": UUID,
    "char_offset": int,        # split point in the text (0 < offset < len(text))
    "session_id": UUID,
    "client_corr_id": UUID,    # idempotency
}
```

**Algorithm.**
1. `SELECT id, session_id, speaker_id, text, start_ms, end_ms, display_order, content_hash FROM segments WHERE id = :sid AND session_id = :session_id FOR UPDATE` — row lock, scoped by session_id (defense in depth).
2. Validate `0 < char_offset < len(text)`. Reject at boundaries (no zero-length halves).
3. Compute split:
   - `text_a = text[:char_offset].rstrip()`
   - `text_b = text[char_offset:].lstrip()`
   - `duration = end_ms - start_ms`
   - `ratio = len(text_a) / (len(text_a) + len(text_b))` (proportional by char count, since true timing is in `word_alignment`)
   - `split_ms = start_ms + int(duration * ratio)`
   - `a: (start_ms, split_ms)`, `b: (split_ms, end_ms)`
4. Recompute `content_hash` for both halves (SHA256 of normalized text, matching existing `_compute_content_hash` in `app/utils/text_hash.py`).
5. Shift all `segments.display_order` rows in this session with `display_order >= original.display_order + 1` UP by 1 (so the new row fits at `original.display_order + 1`).
6. `UPDATE segments SET text=:text_a, end_ms=:split_ms, content_hash=:hash_a WHERE id=:original_id`.
7. `INSERT INTO segments (id, session_id, speaker_id, text, start_ms, end_ms, display_order, content_hash) VALUES (gen_random_uuid(), :sid, :speaker_id, :text_b, :split_ms, :end_ms, :order_b, :hash_b) RETURNING id` — new row inherits `speaker_id`.
8. **`word_alignment` redistribution.**
   - `SELECT id, segment_id, word, start_ms, end_ms FROM word_alignment WHERE segment_id = :original_id ORDER BY start_ms`.
   - For each row, if `end_ms <= split_ms` → keep `segment_id = original_id`. Else `UPDATE word_alignment SET segment_id = :new_id WHERE id = :wa_id`.
   - Edge: a word that straddles the split boundary — its `start_ms < split_ms < end_ms` — is assigned to whichever half contains `start_ms`. We do NOT split words; karaoke alignment stays intact.
9. **`kp_annotations` (key-point anchors) policy.**
   - Existing rows: `kp_annotations(id, segment_id, kp_offset_start, kp_offset_end, label, ...)` where offsets are character offsets into the segment's text.
   - For each annotation on the original segment:
     - If `kp_offset_end <= char_offset` → stays on the original (now `text_a`).
     - If `kp_offset_start >= char_offset` → reassign to new segment with offsets shifted: `(kp_offset_start - char_offset, kp_offset_end - char_offset)`.
     - If the annotation straddles the boundary → **drop** the annotation and emit an `audit_events` row `kind='kp.dropped_on_split'` with the original offsets. Default behavior; user-confirmation UI lives in Phase 4.
10. Return `{"action": "split", "original_segment_id": ..., "new_segment_id": ..., "dropped_kp_count": N}`.

**Idempotency.** Wrapped in the advisory lock + a `client_corr_id`-based dedup check against `corrections` ledger before execution: if a row with this `client_corr_id` already exists for this session, return the cached result without re-executing.

### 3.4 Merge executor (`app/services/segment_merge.py`)

**Input** (`MergeBody`):
```python
{
    "correction_type": "merge",
    "segment_id": UUID,        # the segment to merge INTO (will keep its id)
    "session_id": UUID,
    "client_corr_id": UUID,
}
```

The frontend passes only the LEFT segment — we look up its right-neighbor by `display_order = left.display_order + 1` in the same session.

**Algorithm.**
1. Lock both rows: `SELECT … FROM segments WHERE id = :sid AND session_id = :session_id FOR UPDATE` for the left; second `SELECT` for the row with `session_id = :session_id AND display_order = left.display_order + 1 FOR UPDATE`.
2. Validate the right neighbor exists — reject `MERGE_NO_NEIGHBOR` if `left` is the last segment.
3. Validate `left.speaker_id == right.speaker_id`. Reject `MERGE_SPEAKER_MISMATCH` otherwise (no implicit cross-speaker merge).
4. `merged_text = (left.text.rstrip() + ' ' + right.text.lstrip())` — single space separator, no double-space if either side has trailing/leading whitespace.
5. `merged_end_ms = right.end_ms` (left's start_ms stays).
6. Recompute `content_hash` from `merged_text`.
7. `UPDATE segments SET text=:merged, end_ms=:merged_end, content_hash=:hash WHERE id=:left_id`.
8. `UPDATE word_alignment SET segment_id=:left_id WHERE segment_id=:right_id` (reparent all right's alignment rows).
9. `kp_annotations` policy on merge: shift right-side annotations by `+len(left.text.rstrip() + ' ')` to keep their offsets aligned in the merged text.
10. `DELETE FROM segments WHERE id=:right_id` (cascades to nothing — `word_alignment` was reparented; `kp_annotations` was reparented).
11. Compact `display_order`: shift everything after the deleted row DOWN by 1.
12. Return `{"action": "merge", "kept_segment_id": :left_id, "deleted_segment_id": :right_id}`.

### 3.5 Error envelopes (reuse existing `CORRECTION_*` family)

| Code | When |
|---|---|
| `SPLIT_INVALID_OFFSET` | `char_offset <= 0` or `>= len(text)` |
| `SPLIT_SEGMENT_NOT_FOUND` | `segment_id` not in session |
| `MERGE_NO_NEIGHBOR` | left is the last segment |
| `MERGE_SPEAKER_MISMATCH` | left and right have different `speaker_id` |
| `CORRECTION_DUP_CLIENT_ID` (existing) | `client_corr_id` already in ledger — return cached result |
